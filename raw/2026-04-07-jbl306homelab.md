## Service Inventory

Remote access is provided via [Cloudflare Tunnel](docs/08-cloudflare-tunnel.md) — no ports forwarded, no public IP exposed. LAN access uses AdGuard DNS rewrites (`*.jbl-lab.com → 192.168.1.238`) over HTTP to Caddy on port `80`; internal wildcard TLS is intentionally not used. DNS rewrites are managed via `config/adguard/dns-rewrites.json` and synced with `scripts/ops/sync-dns-rewrites.sh`.

## Quick Start

### 1. Server Setup

Follow the docs in order:

1. [Ubuntu Server Setup](docs/01-ubuntu-setup.md)
2. [Docker Installation](docs/02-docker-install.md)
3. [Storage Setup](docs/03-storage-setup.md)
4. [Tailscale VPN](docs/04-tailscale-setup.md)
5. [NAS Setup](docs/07-nas-setup.md) *(optional — when NAS is purchased)*

### 2. Configure

```bash
# Clone to the server
git clone <your-repo-url> /opt/homelab
cd /opt/homelab

# Create environment file
cp .env.example .env
nano .env  # Fill in your values

# Run initial setup (creates directories + permissions)
find scripts -type f -name '*.sh' -exec chmod +x {} +
./scripts/ops/setup.sh
```

### 3. Deploy

```bash
# Deploy everything
./scripts/ops/deploy.sh

# Or deploy specific stacks (run from project root)
docker compose -f compose/compose.proxy.yml --env-file .env up -d
docker compose -f compose/docker-compose.yml --env-file .env logs -f caddy
```

> Always run from the project root (`~/projects/homelab`) so `.env` is found automatically via `--env-file .env`.

> The default root stack excludes `nba-ml`. Deploy it separately after setting `NBA_ML_ENGINE_PATH` in `.env` and ensuring the `nba-ml-engine` repo exists at that path.

#### 3.5 Check Status After Run
```bash
docker ps --format 'table {{.Names}}\t{{.Status}}\t{{.Ports}}'

# Container count
docker ps -q | wc -l
```

### 4. Post-Deploy

See [Service Configuration Guide](docs/05-service-guide.md) for per-service setup instructions.

## GitHub Actions

The repository includes two GitHub Actions workflows:

| Workflow | Trigger | Purpose |
|------|--------|---------|
| `deploy.yml` | Pull requests, pushes to `main`, manual dispatch | Validates the full merged Compose config. On `main` pushes and manual runs, deploys to the server over SSH. |
| `release.yml` | Git tags matching `v*.*.*` | Validates the Compose config, packages the deployable repo contents, and creates a GitHub Release with attached assets. |

Configure these in GitHub under Settings → Secrets and variables → Actions:

| Secret | Required | Description |
|--------|----------|-------------|

| `HOMELAB_SSH_USERNAME` | Yes | SSH user used for deployments |
| `HOMELAB_SSH_PRIVATE_KEY` | Yes | Private key matching an authorized key on the server |
| `HOMELAB_SSH_FINGERPRINT` | Recommended | SSH host public key fingerprint for host verification |

### Optional repository variables

| Variable | Default | Description |

| `HOMELAB_SSH_PORT` | `22` | SSH port for the server |

### Deployment behavior

1. Pushes to `main` validate the Compose stack on GitHub Actions and then SSH into the server.
2. The deploy job runs `git pull --ff-only origin main` inside the server checkout.
3. It then runs [scripts/ops/deploy.sh](scripts/ops/deploy.sh), deploying all stacks by default.
4. The deploy script recreates missing top-level media config directories for Plex, Jellyfin, and Seerr and resets their directory owner to `PUID:PGID` before `docker compose up`, which prevents empty or root-owned bind mounts from breaking service startup.
5. Manual workflow runs can optionally limit deployment to specific stacks using a space-separated value such as `proxy media riven`.

### Release behavior

Create and push a tag such as `v1.0.0` to trigger a GitHub Release:

```bash
git tag v1.0.0
git push origin v1.0.0
```

The release attaches:

1. `homelab-vX.Y.Z.tar.gz` — deployable repo snapshot
2. `homelab-vX.Y.Z.tar.gz.sha256` — checksum for verification

## Architecture

```
Internet → Cloudflare Edge (TLS termination)
              │
              ▼
         cloudflared (tunnel connector)
              │
              ▼
         Caddy (:80 HTTP)
              │
    ┌─────────┼─────────┬──────────┬──────────┬──────────┬──────────┬────────────┬──────────┬────────┐
    │         │         │          │          │          │          │            │          │        │
  [media]   [riven]  [stremio]    [infra]    [cloud]    [photos]  [nba-ml]  [monitoring]   [wiki]

LAN → AdGuard DNS (*.jbl-lab.com → 192.168.1.238) → Caddy (:80)
```

**Remote access**: Cloudflare terminates public TLS → tunnel → `cloudflared` → Caddy `:80` → service (Plex exception: tunnel → `host.docker.internal:32400` directly).

**LAN access**: AdGuard DNS rewrites `*.jbl-lab.com` → `192.168.1.238`. HTTP on `:80` routes to service-specific Caddy site blocks. Services requiring TLS from LAN (`torrentio`, `vault`) get more-specific DNS rewrites pointing to Cloudflare proxy IPs — exact matches override the wildcard. See `config/adguard/dns-rewrites.json` for the current rewrite config.

### Resource Limits

All containers have CPU and memory limits configured via `deploy.resources.limits` to prevent runaway processes from starving the host (i9-13900HK, 20 threads, 32GB RAM). Key allocations:

| Service | Memory | CPU | Rationale |
|---------|--------|-----|-----------|
| Plex / Jellyfin | 4G | 4.0 | Media transcoding (iGPU handles most, CPU fallback) |
| nba-ml-api | 4G | 4.0 | ML training jobs via scheduler (capped to prevent runaway) |
| Riven | 2G | 2.0 | Media management + VFS operations |
| MLflow | 2G | 1.0 | Experiment tracking + artifact storage |
| Knightcrawler Producer | 512M | 0.5 | Torrent indexer (.NET, periodic scraping bursts); DMM crawler disabled because upstream hashlist ingestion exceeds the host-safe memory envelope |
| Databases (PG/Maria) | 512M–2G | 0.5–1.0 | Scaled per workload |
| Redis instances | 64M–512M | 0.25–0.5 | Cache-only, low footprint |
| Wiki Ingest API | 256M | 0.5 | FastAPI capture service (low traffic, light workload) |
| Infrastructure | 64M–256M | 0.25–0.5 | Caddy, AdGuard, Homepage, etc. |
| Monitoring stack | 64M–512M | 0.25–1.0 | Prometheus, Grafana, cAdvisor, Node Exporter |

| Group | Services |
|-------|----------|
| **Media** | Plex, Jellyfin, Seerr, Riven, Knightcrawler |

Monitors use Docker container names for intra-network health checks (see `scripts/monitoring/update_uptime_kuma.py`).

### Push Notifications (ntfy)

All service alerts are pushed to [ntfy.sh](https://ntfy.sh) for real-time mobile/desktop notifications.

**Two notification layers:**

| Layer | Source | What it catches |
|-------|--------|-----------------|
| **Uptime Kuma → ntfy** | HTTP health checks | Service down/up, response errors, timeouts |
| **Docker event watcher → ntfy** | Docker engine events | Container crashes (non-zero exit), OOM kills, unhealthy status, restart loops |

**Setup:**
1. Subscribe to your topic in the [ntfy app](https://ntfy.sh) (topic configured in `.env` as `NTFY_TOPIC`)
2. Uptime Kuma notifications are managed by `scripts/monitoring/update_uptime_kuma.py`
3. The `docker-notify` container runs in the monitoring stack, watching Docker events

**Configuration** (in `.env`):
```bash
NTFY_SERVER=https://ntfy.sh
NTFY_TOPIC=your-homelab-alerts
```

# Ubuntu Server 24.04 LTS — Initial Setup

Guide for setting up a fresh Ubuntu Server 24.04 LTS installation on the Beelink GTi13 Ultra.

## 1. Install Ubuntu Server

1. Download Ubuntu Server 24.04 LTS ISO from https://ubuntu.com/download/server
2. Flash USB with [Ventoy](https://www.ventoy.net/) and copy the ISO to the drive.
3. Boot the Beelink from USB (press `F7` during POST for boot menu)
4. Choose one of these install methods:

### Option A — Guided install with keyboard/monitor

Follow the guided installer:
- Language: English
- Keyboard: your layout
- Network: DHCP (assign static IP on your router later)
- Storage: Use entire disk (1TB NVMe)
- Profile: Create your user (e.g., `josh`)
- SSH: **Install OpenSSH server** ← important
- Snaps: Skip all

### Option B — Hands-off install with Ventoy autoinstall (no keyboard or mouse)

Use Ubuntu Server's autoinstall feature with a [Ventoy](https://www.ventoy.net/) USB drive so the Beelink can install unattended.

**1. Prepare the USB drive**

Format a USB drive with Ventoy, then arrange files so it looks like this:

```
USB Drive (Ventoy)
├── ubuntu-24.04-live-server-amd64.iso
└── ventoy/
    ├── ventoy.json
    └── user-data
```

**2. Create `ventoy/ventoy.json`**

This tells Ventoy to feed your `user-data` file to the Ubuntu installer as an autoinstall config:

```json
{
  "auto_install": [
    {
      "image": "/ubuntu-24.04-live-server-amd64.iso",
      "template": "/ventoy/user-data"
    }
  ]
}
```

**3. Create `ventoy/user-data`**

Generate a hashed password first (run on any Linux/macOS machine or WSL):

```bash
openssl passwd -6
# Enter your desired password at the prompt. Copy the $6$... output.
```

Then create the autoinstall config:

```yaml
#cloud-config
autoinstall:
  version: 1
  interactive-sections: []
  identity:
    hostname: beelink
    username: josh
    password: "$6$paste-your-hashed-password-here"
  ssh:
    install-server: true
    allow-pw: true
  storage:
    layout:
      name: direct
```

> **Network note:** The config above omits the `network` block so cloud-init auto-detects the active interface via DHCP. If you need to pin a specific interface, add a `network` block — but the interface name varies by hardware (e.g., `enp1s0`, `enp2s0`, `eno1`). Check with `ip link` after first boot.

> **interactive-sections:** Setting this to `[]` prevents Ubuntu 24.04 from pausing to ask for confirmation during autoinstall. Without it, the installer may wait for a keypress.

**4. Boot and install**

1. Insert the USB drive into the Beelink.
2. Boot from USB (press `F7` during POST for the one-time boot menu, or set USB as first boot device in BIOS).
3. Select the Ubuntu ISO from the Ventoy menu.
4. The installer runs unattended — no keyboard or monitor needed after this point.
5. When complete, the machine reboots. Remove the USB drive.
6. Connect over SSH from another machine: `ssh josh@<server-ip>`

> For a fully headless setup, configure the Beelink BIOS to boot from USB automatically so you never need a monitor. After install, it will boot from the NVMe on its own.

# Tailscale VPN Setup

Set up Tailscale for secure remote access to your homelab without port forwarding.

## Why Tailscale

- **Zero config VPN**: No port forwarding, no dynamic DNS, no firewall holes
- **MagicDNS**: Access services via `beelink.tailnet-name.ts.net` from any device
- **Subnet routing**: Expose your entire LAN to Tailscale-connected devices
- **Free tier**: Covers personal use (up to 100 devices, 3 users)

## 1. Install Tailscale on the Server

```bash
# One-line install
curl -fsSL https://tailscale.com/install.sh | sh

# Start and authenticate
sudo tailscale up

# Follow the URL printed to authenticate with your Tailscale account
```

## 2. Enable Subnet Routing (Optional)

Allow Tailscale-connected devices to access your entire LAN:

```bash
# Enable IP forwarding
echo 'net.ipv4.ip_forward = 1' | sudo tee -a /etc/sysctl.d/99-tailscale.conf
echo 'net.ipv6.conf.all.forwarding = 1' | sudo tee -a /etc/sysctl.d/99-tailscale.conf
sudo sysctl -p /etc/sysctl.d/99-tailscale.conf

# Advertise your LAN subnet
sudo tailscale up --advertise-routes=192.168.1.0/24 --accept-routes
```

Then approve the subnet route in the [Tailscale admin console](https://login.tailscale.com/admin/machines):
1. Find your server (beelink)
2. Click "..." → "Edit route settings"
3. Enable the `192.168.1.0/24` route

MagicDNS gives you hostnames like `beelink.tailnet-name.ts.net`:

1. Go to [Tailscale DNS settings](https://login.tailscale.com/admin/dns)
2. Enable **MagicDNS**
3. Your server is now accessible at `beelink` from any Tailscale-connected device

## 4. Install Tailscale on Your Devices

Install on any device you want to access the homelab from:

- **Windows**: Download from https://tailscale.com/download
- **macOS**: App Store or https://tailscale.com/download
- **iOS/Android**: App Store / Play Store

## FlareSolverr + Galloping Bot

The jobs stack now includes an internal-only FlareSolverr service used to pre-solve the EZLinks Cloudflare challenge for Galloping Bot.

### How it works

- The cron helper starts at 11:30 AM ET, but the FlareSolverr solve does not happen immediately.
- The actual FlareSolverr request happens when Galloping Bot begins its pre-warm window about 5 minutes before release, so for a noon release it starts around 11:55 AM ET.
- FlareSolverr requests `https://unioncountygolf.ezlinksgolf.com/index.html` and solves the interactive challenge once.
- Galloping Bot imports the returned `cf_clearance` and related cookies into its nodriver browser session.
- After that import, Galloping Bot still uses its existing browser-backed XHR flow for `GET /api/search/init`, `POST /api/search/search`, and the booking/cart endpoints.
- The first live tee-time search still waits until the release time plus `GALLOPING_SNIPE_DELAY`, so with the default 2-second delay the first real search happens at about 12:00:02 PM ET.

This is important: direct cookie replay with a plain HTTP client still gets blocked on EZLinks. The supported pattern is FlareSolverr cookie prewarm plus browser XHRs from Galloping Bot.

### Validate the helper service

```bash
docker compose -f compose/compose.jobs.yml --env-file .env up -d flaresolverr
docker compose -f compose/compose.jobs.yml --env-file .env logs -f flaresolverr
curl http://localhost:8191/
```

The readiness endpoint should return JSON similar to `{"msg":"FlareSolverr is ready!"}` if you temporarily publish the port for debugging. In the committed compose file the service stays internal-only and is reached as `http://flaresolverr:8191/v1` from the Galloping Bot container.

### Smoke-test the booking flow

```bash
docker compose -f compose/compose.jobs.yml --env-file .env run --rm galloping-bot --discover
```

If the prewarm is working, the bot should clear Cloudflare, call `GET /api/search/init`, and print the live EZLinks course list without manual intervention.

## Labs-Wiki Free-Tier URL Backfill

**Stack:** `jobs` | **Execution:** host cron wrapper | **Schedule:** hourly until complete

This helper runs the main `labs-wiki` checkout conservatively against the GitHub Models free tier. Each invocation processes at most one URL raw, prioritizes the high-value score lane first, then continues into the full remainder once the high-value queue is empty.

### How it works

- The host wrapper lives at `scripts/ops/labs-wiki-backfill.sh`.
- It reads `.env`, uses the root `labs-wiki` checkout by default, and honors `LABS_WIKI_BACKFILL_PATH` only when you intentionally want a different checkout/worktree.
- It reuses `GITHUB_MODELS_TOKEN`.
- If host `python3` is missing ingest dependencies, it bootstraps a local venv at `${HOMELAB_BASE}/data/labs-wiki-backfill/.venv`.
- Every run writes queue state under `${HOMELAB_BASE}/data/labs-wiki-backfill/`.
- If GitHub Models returns a free-tier limit signal, the wrapper sends an ntfy and exits so the next hourly cron window can retry.
- When both the high-value and full queues are empty, it sends a single completion ntfy and stays quiet on later cron runs.

### Manual status / smoke test

```bash
./scripts/ops/labs-wiki-backfill.sh --status
./scripts/ops/labs-wiki-backfill.sh --high-value
```

Use `--status` to inspect the remaining high/full queues without spending quota. `--high-value` or `--full` force a single-item run from that lane.

### Cron install

```bash
5 * * * * /opt/homelab/scripts/ops/labs-wiki-backfill.sh >> /opt/homelab/logs/labs-wiki-backfill.log 2>&1
```

That schedule runs once per hour around the clock until the queue is empty. The wrapper remains quiet after completion, so leaving the cron entry installed is safe.

## Riven (Media Management + Real-Debrid)

Riven handles content discovery (via MDBList, Trakt, Seerr), torrent scraping (via Knightcrawler), debrid integration (Real-Debrid API), and provides a FUSE virtual filesystem (RivenVFS) that streams directly from Real-Debrid CDN.

### Architecture

```
Seerr (user requests) + MDBList/Trakt (auto-discovery)
  → Riven (content management)
    → Knightcrawler (torrent scraping)
      → Real-Debrid (cache/download)
        → RivenVFS (FUSE mount at /mnt/riven)
          → Plex / Jellyfin (library scan → click to stream)
```

### Prerequisites: Shared bind mount

RivenVFS requires the host mount point to be a shared bind mount for FUSE propagation into other containers (Plex/Jellyfin). The `setup.sh` script handles this, but it must persist across reboots.

**Option A — fstab entry** (recommended):

```bash
# Add to /etc/fstab
/mnt/riven  /mnt/riven  none  bind,rshared  0  0
```

**Option B — systemd one-shot unit**:

```ini
# /etc/systemd/system/riven-bind-shared.service
[Unit]
Description=Make Riven data bind mount shared
After=local-fs.target
Before=docker.service

[Service]
Type=oneshot
ExecStart=/usr/bin/mount --bind /mnt/riven /mnt/riven
ExecStart=/usr/bin/mount --make-rshared /mnt/riven
RemainAfterExit=yes

[Install]
WantedBy=multi-user.target
```

Enable: `sudo systemctl enable --now riven-bind-shared.service`

**Verify propagation**:

```bash
findmnt -T /mnt/riven -o TARGET,PROPAGATION  # expect: shared or rshared
```

### First-run setup

1. Deploy the stack:
   ```bash
   ./scripts/ops/deploy.sh riven
   ```

2. Open the Riven frontend at `http://<server-ip>:8070`

3. **Register an account**: Click "Register", create a username/password. The frontend uses `better-auth` with a local SQLite database (`/riven/config/auth.db`) for session management.

4. **Log in** and navigate to Settings.

5. Configure **Debrid settings**:
   - Provider: Real-Debrid
   - API Key: your RD API token (from https://real-debrid.com/apitoken)

6. Configure **Content services** (where Riven discovers what to download):
   - **Overseerr**: URL `http://seerr:5055`, API key from Seerr → Settings → General
   - **MDBList** (optional): Add lists for auto-populated trending/popular content
   - **Trakt** (optional): Connect for watchlist sync

7. Configure **Scraping services** (where Riven finds torrents):
   - **Knightcrawler**: URL `http://knightcrawler-addon:7000` (Docker DNS, same stremio network)
   - **Torrentio**: URL `https://torrentio.strem.fun` (public addon, rate-limited)

8. Configure **Media server**:
   - **Jellyfin**: URL `http://jellyfin:8096`, API key from Jellyfin → Dashboard → API Keys
   - **Plex** (optional): URL `http://<server-ip>:32400`, Plex token

9. Verify the VFS mount:
   ```bash
   ls /mnt/riven/
   # Should show: movies/ shows/ (etc.) after Riven processes first items
   ```

### Requesting media (end-to-end flow)

**Option A — Via Riven frontend**:
1. Open `http://riven.jbl-lab.com` (or `http://<server-ip>:8070`)
2. Log in with your account
3. Use the search bar to find a movie/show by title
4. Click the result → click "Request" to add it to Riven's queue
5. Riven automatically: scrapes Knightcrawler/Torrentio → sends to Real-Debrid → exposes via VFS → updates Jellyfin

**Option B — Via Seerr (Overseerr)**:
1. Open `http://seerr.jbl-lab.com` (or `http://<server-ip>:5055`)
2. Search for a movie/show, click "Request"
3. Riven polls Seerr's API and picks up the request automatically

**Option C — Via API** (for automation):
```bash
curl -X POST "http://<server-ip>:8070/api/v1/items/add" \
  -H "Content-Type: application/json" \
  -H "x-api-key: $RIVEN_API_KEY" \
  -d '{"tmdb_ids": ["550"], "media_type": "movie"}'
```

**Monitoring progress**: The Riven frontend browse page shows all items with their current state (Requested → Scraped → Downloaded → Completed).

### Environment variables (frontend)

| Variable | Required | Description |
|----------|----------|-------------|
| `ORIGIN` | Yes | Public URL of the frontend (for CSRF protection) |
| `BACKEND_URL` | Yes | Internal URL to the Riven backend |
| `BACKEND_API_KEY` | Yes | Riven backend API key (from `RIVEN_API_KEY` in `.env`) |

| `ENABLE_EMAIL_PASSWORD_SIGNUP` | No | Set to `true` to allow email/password registration |
| `DISABLE_PLEX` | No | Set to `true` to hide Plex login option |

### How the pipeline works

```
User requests via Seerr → Riven polls Overseerr API
  → Riven scrapes Knightcrawler for torrent matches
    → Riven sends best match to Real-Debrid
      → RivenVFS exposes file at /mnt/riven/movies/Title (Year)/
        → Plex/Jellyfin sees it via rslave mount propagation
          → User clicks poster → streams from RD CDN
```

### Plex library paths

Riven's VFS directory structure:

| Riven VFS path | Plex library type |
|---------------|-------------------|
| `/data/movies` | Movies |
| `/data/shows` | TV Shows |
| `/data/anime_movies` | Movies (anime) |
| `/data/anime_shows` | TV Shows (anime) |

### Jellyfin library paths

Same as Plex — use `/data/movies`, `/data/shows`, etc.

### Troubleshooting

- **Mount empty after Riven restart**: Verify host bind mount is `rshared` (`findmnt -T /mnt/riven -o PROPAGATION`). Clear stale mount: `sudo fusermount -uz /mnt/riven`
- **Knightcrawler not found**: Ensure Riven and Knightcrawler share the `stremio` Docker network. Check with `docker network inspect stremio`.
- **Plex/Jellyfin shows empty library**: Verify rslave propagation inside the container: `docker exec -it jellyfin sh -c 'findmnt -T /data -o PROPAGATION'` — should show `slave` or `rslave`.
- **Port 8070 conflict**: Riven runs internally on 8080, mapped to host 8070 to avoid conflict with Nextcloud (8080).

# NAS Setup (Optional — Post-Deploy)

Configure NAS mounts for media storage, Immich photo offload, Nextcloud data, and backup targets. This is optional — all services run without a NAS using local SSD storage.

> **When to do this:** After your NAS hardware is purchased and on the network. The homelab runs fine without it.

## Prerequisites

- NAS accessible on your LAN (e.g., Synology, TrueNAS, or any NFS/SMB server)
- NAS IP address and share paths
- NFS or SMB credentials configured on the NAS

## 1. Install Mount Utilities

These should already be installed from the Ubuntu setup, but verify:

```bash
sudo apt install -y nfs-common cifs-utils
```

## 2. Create Mount Points

```bash
sudo mkdir -p /mnt/nas/media/{movies,tv,music,books}
sudo mkdir -p /mnt/nas/backups
sudo mkdir -p /mnt/nas/photos
```

## 3. Mount the NAS

### Option A: NFS Mount (recommended for Linux NAS)

```bash
# Test mount
sudo mount -t nfs <nas-ip>:/volume1/media /mnt/nas/media

# If it works, add to /etc/fstab for persistence
echo '<nas-ip>:/volume1/media /mnt/nas/media nfs defaults,_netdev,soft,timeo=150 0 0' | sudo tee -a /etc/fstab
echo '<nas-ip>:/volume1/backups /mnt/nas/backups nfs defaults,_netdev,soft,timeo=150 0 0' | sudo tee -a /etc/fstab
echo '<nas-ip>:/volume1/photos /mnt/nas/photos nfs defaults,_netdev,soft,timeo=150 0 0' | sudo tee -a /etc/fstab

# Mount all
sudo mount -a
```

### Option B: SMB/CIFS Mount (for Windows shares or Synology)

```bash
# Create credentials file (more secure than inline)
sudo tee /etc/nas-credentials > /dev/null << 'EOF'
username=your_nas_user
password=your_nas_password
EOF
sudo chmod 600 /etc/nas-credentials

# Add to /etc/fstab
echo '//<nas-ip>/media /mnt/nas/media cifs credentials=/etc/nas-credentials,uid=1000,gid=1000,_netdev,iocharset=utf8 0 0' | sudo tee -a /etc/fstab
echo '//<nas-ip>/backups /mnt/nas/backups cifs credentials=/etc/nas-credentials,uid=1000,gid=1000,_netdev,iocharset=utf8 0 0' | sudo tee -a /etc/fstab
echo '//<nas-ip>/photos /mnt/nas/photos cifs credentials=/etc/nas-credentials,uid=1000,gid=1000,_netdev,iocharset=utf8 0 0' | sudo tee -a /etc/fstab

# Mount all
sudo mount -a

# Cloudflare Tunnel Setup

Expose homelab services securely via Cloudflare Tunnel — no port forwarding, no public IP exposure.

## How It Works

```
Internet → Cloudflare Edge (TLS + DDoS + WAF) → Tunnel → cloudflared container → Caddy :80 → service
Internet → Cloudflare Edge → Tunnel → cloudflared container → host.docker.internal:32400 (Plex)
LAN      → AdGuard DNS rewrite → Caddy :80 (HTTP) → service
```

- **Remote**: Cloudflare terminates TLS, forwards traffic through an encrypted tunnel to `cloudflared`, which proxies to Caddy on port 80

## Step 5: AdGuard DNS Rewrites (LAN Access)

DNS rewrites let `*.jbl-lab.com` resolve to the server's LAN IP so LAN clients hit Caddy directly (HTTP) instead of routing through Cloudflare.

### Strategy

| Rewrite | Answer | Purpose |
|---------|--------|---------|
| `*.jbl-lab.com` | `192.168.1.238` | Wildcard — all subdomains resolve to LAN server via Caddy HTTP |
| `jbl-lab.com` | `192.168.1.238` | Apex domain — same as wildcard |
| `torrentio.jbl-lab.com` | Cloudflare proxy IPs | Stremio addon — clients require HTTPS for addon URLs |
| `vault.jbl-lab.com` | Cloudflare proxy IPs | Vaultwarden — WebAuthn/FIDO2 and Bitwarden clients require HTTPS |
| `photos.jbl-lab.com` | Cloudflare proxy IPs | Immich mobile/web clients should keep the public HTTPS route on LAN too |

Most services use the wildcard and go **LAN → Caddy → Service** (fast, local). Services that need TLS from LAN get more-specific rewrites pointing to Cloudflare's proxy IPs (exact matches override the wildcard).

### Automated sync

The desired state is defined in `config/adguard/dns-rewrites.json`. Sync it to AdGuard via:

```bash

./scripts/ops/sync-dns-rewrites.sh --dry-run

# Apply changes

```

The script uses the AdGuard REST API (`http://localhost:3000`) and is idempotent — safe to run repeatedly. Set `ADGUARD_URL`, `ADGUARD_USER`, `ADGUARD_PASS` env vars if credentials or URL differ.

### Manual setup (alternative)

1. Open AdGuard: `http://192.168.1.238:3000`
2. Go to **Filters → DNS rewrites**
3. Add the wildcard: `*.jbl-lab.com` → `192.168.1.238`
4. Add the apex: `jbl-lab.com` → `192.168.1.238`
5. For Cloudflare overrides, resolve current proxy IPs first:

```

6. Add each Cloudflare IP as a separate rewrite for `torrentio.jbl-lab.com`, `vault.jbl-lab.com`, and `photos.jbl-lab.com`

### When to add a Cloudflare override

Add a more-specific rewrite (domain → Cloudflare IPs) when a client **requires HTTPS** and won't work over plain HTTP:

- **Stremio addons** — the app validates addon URLs are HTTPS
- **Vaultwarden** — WebAuthn/FIDO2 requires a secure context; Bitwarden clients enforce HTTPS for non-localhost

## Troubleshooting

### Tunnel shows "disconnected" in Cloudflare dashboard

```bash
docker logs cloudflared --tail 50
```

- **Token invalid**: Regenerate in Cloudflare dashboard, update `.env`, redeploy
- **DNS resolution failure**: Check the server can resolve `api.cloudflare.com`

### Hostname is configured in the tunnel but does not resolve publicly

If `cloudflared` shows the hostname in its ingress config but `curl https://subdomain.jbl-lab.com` fails with `Could not resolve host`, the problem is at Cloudflare DNS/public-hostname level rather than Caddy.

- Verify the public hostname exists on the tunnel and targets `http://caddy:80`
- Confirm the hostname is attached to the correct `jbl-lab.com` zone
- Check that Cloudflare created the proxied DNS record for the hostname
- Wait for propagation, then retest from a non-LAN client

### Remote access returns 502

The `cloudflared` container can't reach Caddy:

```bash
# Verify both are on the proxy network
docker network inspect proxy | grep -A2 "cloudflared\|caddy"
```

### LAN access to subdomains doesn't work

- Verify AdGuard DNS rewrites are set (Filters → DNS rewrites)
- Verify the device is using AdGuard as its DNS server (`nslookup jellyfin.jbl-lab.com`)
- Verify exact-host overrides beat the wildcard (`dig +short @192.168.1.238 nba-dashboard.jbl-lab.com`)
- LAN access is HTTP only. Cloudflare Tunnel provides real TLS for public (remote) access

### Service works on IP:port but not via subdomain

The Caddy label may be misconfigured. Check the generated Caddyfile:

```bash
docker logs caddy 2>&1 | grep -i "error\|caddyfile"
```

# Knightcrawler Operations Guide

How to monitor, query, and troubleshoot the self-hosted Knightcrawler (Torrentio alternative) stack.

**Version**: 2.0.28 (all services)

---

## Architecture Overview

Knightcrawler runs as 6 active containers (debridcollector is stopped):

| Container | Role | Status |
|-----------|------|--------|
| `knightcrawler-addon` | Stremio-compatible API — serves stream results at port 7000 | Active |
| `knightcrawler-producer` | Crawls torrent sources on schedule, publishes to message queue | Active |
| `knightcrawler-consumer` | Consumes from queue, parses torrents via RTN, writes to `torrents` table | Active |
| `knightcrawler-debridcollector` | ~~Queries Real-Debrid instant availability~~ **STOPPED** — RD API disabled | Stopped |
| `knightcrawler-postgres` | PostgreSQL 16 database (user: `postgres`, db: `knightcrawler`, `shm_size: 512m`) | Active |
| `knightcrawler-redis` | Cache layer for addon stream lookups | Active |
| `knightcrawler-lavinmq` | Message broker (RabbitMQ-compatible) for producer → consumer pipeline | Active |

### Data flow

```
Producer (crawls Nyaa, YTS, built-in Torrentio; DMM disabled locally)
  → LavinMQ queue
    → Consumer (parses via RTN, writes torrents + ingested_torrents tables)
      → Cron: kc-populate-files.sh (maps DMM IMDB IDs to files table every 15 min)
      → Cron: kc-scrape-recent.sh (imports current-year gaps from upstream Torrentio every 6 hours)
        → Addon (queries files table, returns streams to Stremio/Riven)
```

> **Key insight**: The `files` table is what the addon queries to return streams. If `files` is empty, the addon returns 0 streams even if `torrents` has data.
>
> **RD API disabled**: Real-Debrid permanently disabled their instant availability endpoint (`error_code: 37`). The debridcollector is stopped because it cannot function. Instead, a cron job (`scripts/knightcrawler/automation/kc-populate-files.sh`) maps DMM torrent data directly to the files table using the IMDB IDs embedded in DMM hashlists and RTN-parsed season/episode data.
>
> **DMM state**: DMM was originally the high-volume ingester and still accounts for most historical `ingested_torrents` rows in this database, but it is currently disabled in `config/knightcrawler/scrapers.json` because the upstream `main.zip` download/extract path is not safe within this host's memory envelope.

### Custom configuration

The producer uses volume-mounted config files instead of the baked-in defaults:

| File | Mount path | Purpose |
|------|-----------|---------|
| `config/knightcrawler/scrapers.json` | `/app/Configuration/scrapers.json` | Crawler source config — disabled broken sources, fixed URLs |
| `config/knightcrawler/torrentio.json` | `/app/Configuration/torrentio.json` | Torrentio proxy crawler config |

### Crawler status

| Crawler | Status | Notes |
|---------|--------|-------|
| **DMM** (DebridMediaManager) | **Disabled locally** | Historically the largest source already present in the DB. Disabled on this host because the producer OOMs on the upstream `main.zip` path at host-safe limits. Restart behavior is page-dedupe, not true cursor resume; see below |
| **Nyaa** | Active | Anime RSS scraper. Working normally |
| **YTS** | Active | Movie RSS. URL fixed from `yts.am` → `yts.bz/rss` in custom config |
| **Torrentio** (proxy) | Active (limited) | Weekly scrape of `torrentio.strem.fun`. Rate-limited to 300 req/hr. **Starts from tt0000001 each cycle** and does not persist durable cursor state in the DB. Supplemented by `kc-scrape-recent.sh` cron for current-year IMDb IDs, which persists its own IMDb cursor file and suppresses recent no-stream candidates |
| **EZTV** | **Disabled** | All mirrors (eztv1.xyz, eztvx.to, eztv.re) return 403 — Cloudflare JS challenge |
| **TPB** (The Pirate Bay) | **Disabled** | apibay.org behind Cloudflare, returns 403 |
| **TGX** (TorrentGalaxy) | **Disabled** | Domain `tgx.rs` dead (DNS failure), `torrentgalaxy.to` unreachable |

### DMM restart semantics

Upstream `DebridMediaManagerCrawler` in v2.0.28 does **not** keep a durable "current page" cursor. Each run:

1. Downloads `main.zip`
2. Extracts every `*.html` file
3. Checks `ingested_pages.url` for the page basename
4. Skips files already marked as ingested
5. Inserts torrents for the remaining pages and then marks those page basenames as ingested

So after a restart, DMM effectively **restarts and skips** rather than **resumes from a stored offset**. The separate `providers` table exists in the schema, but it is not used by DMM here and is empty on this host.

One important edge case: pages that never reach `MarkPageAsIngested` are retried on later runs. That includes crashes before the final write, and pages that decode successfully but end up with zero matched torrents after title parsing.

### Files population crons

Since the debridcollector is broken, two cron jobs populate the `files` table:

```
# Map DMM torrents → files (every 15 min)
*/15 * * * * /home/jbl/projects/homelab/scripts/knightcrawler/automation/kc-populate-files.sh >> /home/jbl/logs/kc-populate.log 2>&1

# Scrape upstream Torrentio for recent content (every 6 hours)
0 */6 * * * /home/jbl/projects/homelab/scripts/knightcrawler/automation/kc-scrape-recent.sh >> /home/jbl/logs/kc-scrape-recent.log 2>&1
```

Homepage's **Scheduled Jobs → KnightCrawler Automation** card is backed by `homepage-db-stats` and shows populate freshness, scrape freshness, current-year titles with files, and a split of missing titles into **No Torrents** vs **Fileless**. The scrape log summary now reports real outcomes (`imported`, `no_streams`, `no_episodes`, `errors`) instead of counting every zero-exit lookup as an import.

#### kc-populate-files.sh

Copies `kc-populate-files.sql` into the postgres container and executes via `psql -f`. Four-pass INSERT into `files` via `torrents` → `ingested_torrents` JOIN:
1. **Movies**: Maps torrent directly to IMDB ID (fileIndex=0)
2. **Series episodes**: Torrents with specific season+episode from RTN parsing (fileIndex=0)
3. **Single-season packs**: Expands packs with one season into per-episode entries using `imdb_metadata_episodes` table (fileIndex=season×1000+episode for idempotency)
4. **Multi-season packs**: Expands packs spanning multiple seasons (e.g., "Show.S01-S04.1080p.BluRay") using `CROSS JOIN LATERAL jsonb_array_elements_text()` on the RTN `season` array, then joins with `imdb_metadata_episodes` for per-episode entries

File titles use `COALESCE(it.rtn_response->>'raw_title', t.title)` — the RTN-parsed release name (e.g., "Peaky.Blinders.S06E05.1080p.AMZN.WEBRip.DDP5.1.x264-TOMMY") instead of the bare IMDB name. The `rtn_response` JSONB also has resolution, codec, quality, and audio metadata.

All passes use `ON CONFLICT DO NOTHING` — safe to re-run.

> **Important**: The script uses `docker cp` + `psql -f` instead of heredoc `<<'SQL'` piped to `docker exec`. The heredoc approach silently hangs when run via cron because `docker exec` stdin handling is unreliable in non-interactive contexts.

Run manually: `/home/jbl/projects/homelab/scripts/knightcrawler/automation/kc-populate-files.sh`

> **Permissions**: All scripts in `scripts/knightcrawler/automation/` must have execute permission (`chmod +x`). If the cron log shows `Permission denied`, the script lost its `+x` bit (e.g., after a `git checkout` or file recreation). Fix: `chmod +x scripts/knightcrawler/automation/*.sh`.

#### kc-scrape-recent.sh (recent Torrentio scraper)

The built-in Torrentio proxy crawler in the producer starts from `tt0000001` each cycle and at 300 req/hr will never reach recent IMDB IDs (tt30000000+). This cron job fills that gap and focuses on the current release year by default:

1. Queries `imdb_metadata` for current-year IMDB IDs that have **no** entries in the `files` table
2. For each ID, calls `kc-import-torrentio.py` which fetches streams from upstream `torrentio.strem.fun`
3. Inserts results into both `torrents` and `files` tables (with ON CONFLICT DO NOTHING)
4. Processes 200 IMDB IDs per run with IMDb-cursor resume (state in `/tmp/kc-scrape-recent.cursor`)
5. Skips series that do not yet have episode metadata in `imdb_metadata_episodes`
6. Persists recent `no_streams` / `no_episodes` outcomes in `automation_scrape_recent_outcomes` so cursor resets do not immediately rescan hopeless candidates

Options:
```bash
kc-scrape-recent.sh                  # Default: current year only, 200 per batch
kc-scrape-recent.sh --year 2025      # Widen search to 2025+
kc-scrape-recent.sh --batch 200      # Larger batch
kc-scrape-recent.sh --type series    # Only TV series
kc-scrape-recent.sh --type movie     # Only movies
```

#### kc-import-torrentio.py (on-demand single-title import)

Import streams for a specific IMDB ID from upstream Torrentio:

```bash
# Import a specific show (auto-detects type from imdb_metadata)
python3 scripts/knightcrawler/automation/kc-import-torrentio.py tt32937780

# Force type and preview SQL without executing

## Database Reference

### Tables

| Table | Purpose | Key columns |
|-------|---------|-------------|
| `torrents` | Parsed torrent metadata | `infoHash`, `title`, `type`, `resolution`, `seeders`, `provider` |
| `files` | RD-available files mapped to IMDB IDs | `infoHash`, `imdbId`, `imdbSeason`, `imdbEpisode`, `title`, `size` |
| `contents` | Raw file paths inside torrents | `infoHash`, `fileIndex`, `path`, `size` |
| `imdb_metadata` | IMDb title dataset (12M+ rows) | `imdb_id`, `title`, `category`, `year` |
| `imdb_metadata_episodes` | Episode-level IMDb data | `parent_id`, season/episode numbers |
| `ingested_torrents` | Raw ingested torrent records | `name`, `source`, `info_hash`, `imdb`, `processed` |
| `ingested_pages` | DMM page-level dedupe state | `url`, `createdAt`, `updatedAt` |
| `automation_scrape_recent_outcomes` | Recent no-stream/no-episode suppression state for `kc-scrape-recent.sh` | `imdb_id`, `last_outcome`, `retry_after`, `attempts` |
| `subtitles` | Subtitle tracks within torrents | `infoHash`, `fileIndex`, `title` |
| `providers` | Reserved crawler cursor table; empty on this host in v2.0.28 | `name`, `lastScraped`, `lastScrapedId` |
| `skip_torrents` | Blocklisted infohashes | `infoHash` |

## Searching for Specific Movies or Shows

### By torrent title (fuzzy text search)

```bash
# Search torrents table for a movie by name
docker exec knightcrawler-postgres psql -U postgres -d knightcrawler -c \
  "SELECT title, type, resolution, seeders, provider
   FROM torrents
   WHERE title ILIKE '%shawshank%'
   ORDER BY seeders DESC NULLS LAST
   LIMIT 20;"

# Search for a TV show
docker exec knightcrawler-postgres psql -U postgres -d knightcrawler -c \
  "SELECT title, type, resolution, seeders, provider
   FROM torrents
   WHERE title ILIKE '%breaking bad%'
   ORDER BY seeders DESC NULLS LAST
   LIMIT 20;"
```

### By IMDB ID

Find the IMDB ID first (from imdb.com URL or metadata table), then query:

```bash
# Look up an IMDB ID from the metadata table
docker exec knightcrawler-postgres psql -U postgres -d knightcrawler -c \
  "SELECT imdb_id, title, category, year
   FROM imdb_metadata
   WHERE title ILIKE '%shawshank%' AND category IN ('movie', 'tvSeries')
   ORDER BY year DESC
   LIMIT 10;"

# Check if files exist for a specific IMDB ID (this is what the addon uses)
docker exec knightcrawler-postgres psql -U postgres -d knightcrawler -c \
  "SELECT f.\"imdbId\", f.title, f.size, t.resolution, t.seeders, t.provider
   FROM files f
   JOIN torrents t ON f.\"infoHash\" = t.\"infoHash\"
   WHERE f.\"imdbId\" = 'tt0111161'
   ORDER BY t.seeders DESC NULLS LAST
   LIMIT 20;"

# For TV shows — search by IMDB ID + season + episode
docker exec knightcrawler-postgres psql -U postgres -d knightcrawler -c \
  "SELECT f.\"imdbId\", f.\"imdbSeason\", f.\"imdbEpisode\", f.title, f.size,
          t.resolution, t.seeders
   FROM files f
   JOIN torrents t ON f.\"infoHash\" = t.\"infoHash\"
   WHERE f.\"imdbId\" = 'tt0903747'
     AND f.\"imdbSeason\" = 1
     AND f.\"imdbEpisode\" = 1
   ORDER BY t.seeders DESC NULLS LAST
   LIMIT 10;"
```

### Check if a torrent has been processed into files

```bash
# Find a torrent and check if it has matching files
docker exec knightcrawler-postgres psql -U postgres -d knightcrawler -c \
  "SELECT t.\"infoHash\", t.title, t.type,
          (SELECT count(*) FROM files f WHERE f.\"infoHash\" = t.\"infoHash\") AS file_count
   FROM torrents t
   WHERE t.title ILIKE '%interstellar%'
   LIMIT 10;"
```

> A torrent with `file_count = 0` means the cron job hasn't run since it was ingested, or it's a non-DMM torrent (no IMDB ID), or it's a series season pack.
