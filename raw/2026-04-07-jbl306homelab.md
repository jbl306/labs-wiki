## Service Inventory

Remote access is provided via [Cloudflare Tunnel](docs/08-cloudflare-tunnel.md) — no ports forwarded, no public IP exposed. LAN access uses AdGuard DNS rewrites (`*.jbl-lab.com → 192.168.1.238`) over HTTP to Caddy on port `80`; internal wildcard TLS is intentionally not used. ## Quick Start

### 1. Server Setup

Follow the docs in order:

1. [Ubuntu Server Setup](docs/01-ubuntu-setup.md)
2. [Docker Installation](docs/02-docker-install.md)
3. [Storage Setup](docs/03-storage-setup.md)
4. [Tailscale VPN](docs/04-tailscale-setup.md)
5. [NAS Setup](docs/07-nas-setup.md) *(optional — when NAS is purchased)*

### 2. Configure

```bash
# Clone to the server
git clone <your-repo-url> /opt/homelab
cd /opt/homelab

# Create environment file
cp .env.example .env
nano .env  # Fill in your values

# Run initial setup (creates directories + permissions)
find scripts -type f -name '*.sh' -exec chmod +x {} +
./scripts/ops/setup.sh
```

### 3. Deploy

```bash
# Deploy everything
./scripts/ops/deploy.sh

# Or deploy specific stacks (run from project root)
docker compose -f compose/compose.proxy.yml --env-file .env up -d
docker compose -f compose/docker-compose.yml --env-file .env logs -f caddy
```

> Always run from the project root (`~/projects/homelab`) so `.env` is found automatically via `--env-file .env`.

> The default root stack excludes `nba-ml`. Deploy it separately after setting `NBA_ML_ENGINE_PATH` in `.env` and ensuring the `nba-ml-engine` repo exists at that path.

#### 3.5 Check Status After Run
```bash
docker ps --format 'table {{.Names}}\t{{.Status}}\t{{.Ports}}'

# Container count
docker ps -q | wc -l
```

### 4. Post-Deploy

See [Service Configuration Guide](docs/05-service-guide.md) for per-service setup instructions.

## GitHub Actions

The repository includes two GitHub Actions workflows:

| Workflow | Trigger | Purpose |
|------|--------|---------|
| `deploy.yml` | Pull requests, pushes to `main`, manual dispatch | Validates the full merged Compose config. On `main` pushes and manual runs, deploys to the server over SSH. |
| `release.yml` | Git tags matching `v*.*.*` | Validates the Compose config, packages the deployable repo contents, and creates a GitHub Release with attached assets. |

### Required repository secrets

Configure these in GitHub under Settings → Secrets and variables → Actions:

| Secret | Required | Description |
|--------|----------|-------------|
| `HOMELAB_SSH_HOST` | Yes | Hostname or IP of the Ubuntu server |
| `HOMELAB_SSH_USERNAME` | Yes | SSH user used for deployments |
| `HOMELAB_SSH_PRIVATE_KEY` | Yes | Private key matching an authorized key on the server |
| `HOMELAB_SSH_FINGERPRINT` | Recommended | SSH host public key fingerprint for host verification |

### Optional repository variables

| Variable | Default | Description |
|----------|---------|-------------|
| `HOMELAB_DEPLOY_PATH` | `/opt/homelab` | Absolute path of the checked-out repo on the server |
| `HOMELAB_SSH_PORT` | `22` | SSH port for the server |

### Deployment behavior

1. Pushes to `main` validate the Compose stack on GitHub Actions and then SSH into the server.
2. The deploy job runs `git pull --ff-only origin main` inside the server checkout.
3. It then runs [scripts/ops/deploy.sh](scripts/ops/deploy.sh), deploying all stacks by default.
4. The deploy script recreates missing top-level media config directories for Plex, Jellyfin, and Seerr and resets their directory owner to `PUID:PGID` before `docker compose up`, which prevents empty or root-owned bind mounts from breaking service startup.
5. Manual workflow runs can optionally limit deployment to specific stacks using a space-separated value such as `proxy media riven`.

### Release behavior

Create and push a tag such as `v1.0.0` to trigger a GitHub Release:

```bash
git tag v1.0.0
git push origin v1.0.0
```

The release attaches:

1. `homelab-vX.Y.Z.tar.gz` — deployable repo snapshot
2. `homelab-vX.Y.Z.tar.gz.sha256` — checksum for verification

## Architecture

```
Internet → Cloudflare Edge (TLS termination)
              │
              ▼
         cloudflared (tunnel connector)
              │
              ▼
         Caddy (:80 HTTP)
              │
    ┌─────────┼─────────┬──────────┬──────────┬──────────┬──────────┬────────────┬──────────┬────────┐
    │         │         │          │          │          │          │            │          │        │
  [media]   [riven]  [stremio]    [infra]    [cloud]    [photos]  [nba-ml]  [monitoring]   [wiki]

LAN → AdGuard DNS (*.jbl-lab.com → 192.168.1.238) → Caddy (:80)
```

**Remote access**: Cloudflare terminates public TLS → tunnel → `cloudflared` → Caddy `:80` → service (Plex exception: tunnel → `host.docker.internal:32400` directly).

**LAN access**: AdGuard DNS rewrites `*.jbl-lab.com` → `192.168.1.238`. HTTP on `:80` routes to service-specific Caddy site blocks. Services requiring TLS from LAN (`torrentio`, `vault`) get more-specific DNS rewrites pointing to Cloudflare proxy IPs — exact matches override the wildcard. See `config/adguard/dns-rewrites.json` for the current rewrite config.

### Resource Limits

All containers have CPU and memory limits configured via `deploy.resources.limits` to prevent runaway processes from starving the host (i9-13900HK, 20 threads, 32GB RAM). Key allocations:

| Service | Memory | CPU | Rationale |
|---------|--------|-----|-----------|
| Plex / Jellyfin | 4G | 4.0 | Media transcoding (iGPU handles most, CPU fallback) |
| nba-ml-api | 4G | 4.0 | ML training jobs via scheduler (capped to prevent runaway) |
| Riven | 2G | 2.0 | Media management + VFS operations |
| MLflow | 2G | 1.0 | Experiment tracking + artifact storage |
| Knightcrawler Producer | 512M | 1.0 | Torrent indexer (.NET, periodic scraping bursts) |
| Databases (PG/Maria) | 512M–2G | 0.5–1.0 | Scaled per workload |
| Redis instances | 64M–512M | 0.25–0.5 | Cache-only, low footprint |
| Wiki Ingest API | 256M | 0.5 | FastAPI capture service (low traffic, light workload) |
| Infrastructure | 64M–256M | 0.25–0.5 | Caddy, AdGuard, Homepage, etc. |

### Push Notifications (ntfy)

All service alerts are pushed to [ntfy.sh](https://ntfy.sh) for real-time mobile/desktop notifications.

**Two notification layers:**

| Layer | Source | What it catches |
|-------|--------|-----------------|
| **Uptime Kuma → ntfy** | HTTP health checks | Service down/up, response errors, timeouts |
| **Docker event watcher → ntfy** | Docker engine events | Container crashes (non-zero exit), OOM kills, unhealthy status, restart loops |

**Setup:**
1. Subscribe to your topic in the [ntfy app](https://ntfy.sh) (topic configured in `.env` as `NTFY_TOPIC`)
2. Uptime Kuma notifications are managed by `scripts/monitoring/update_uptime_kuma.py`
3. The `docker-notify` container runs in the monitoring stack, watching Docker events

**Configuration** (in `.env`):
```bash
NTFY_SERVER=https://ntfy.sh
NTFY_TOPIC=your-homelab-alerts
```

# Ubuntu Server 24.04 LTS — Initial Setup

Guide for setting up a fresh Ubuntu Server 24.04 LTS installation on the Beelink GTi13 Ultra.

## 1. Install Ubuntu Server

1. Download Ubuntu Server 24.04 LTS ISO from https://ubuntu.com/download/server
2. Flash USB with [Ventoy](https://www.ventoy.net/) and copy the ISO to the drive.
3. Boot the Beelink from USB (press `F7` during POST for boot menu)
4. Choose one of these install methods:

### Option A — Guided install with keyboard/monitor

Follow the guided installer:
- Language: English
- Keyboard: your layout
- Network: DHCP (assign static IP on your router later)
- Storage: Use entire disk (1TB NVMe)
- Profile: Create your user (e.g., `josh`)
- SSH: **Install OpenSSH server** ← important
- Snaps: Skip all

### Option B — Hands-off install with Ventoy autoinstall (no keyboard or mouse)

Use Ubuntu Server's autoinstall feature with a [Ventoy](https://www.ventoy.net/) USB drive so the Beelink can install unattended.

**1. Prepare the USB drive**

Format a USB drive with Ventoy, then arrange files so it looks like this:

```
USB Drive (Ventoy)
├── ubuntu-24.04-live-server-amd64.iso
└── ventoy/
    ├── ventoy.json
    └── user-data
```

**2. Create `ventoy/ventoy.json`**

This tells Ventoy to feed your `user-data` file to the Ubuntu installer as an autoinstall config:

```json
{
  "auto_install": [
    {
      "image": "/ubuntu-24.04-live-server-amd64.iso",
      "template": "/ventoy/user-data"
    }
  ]
}
```

**3. Create `ventoy/user-data`**

Generate a hashed password first (run on any Linux/macOS machine or WSL):

```bash
openssl passwd -6
# Enter your desired password at the prompt. Copy the $6$... output.
```

Then create the autoinstall config:

```yaml
#cloud-config
autoinstall:
  version: 1
  interactive-sections: []
  identity:
    hostname: beelink
    username: josh
    password: "$6$paste-your-hashed-password-here"
  ssh:
    install-server: true
    allow-pw: true
  storage:
    layout:
      name: direct
```

> **Network note:** The config above omits the `network` block so cloud-init auto-detects the active interface via DHCP. If you need to pin a specific interface, add a `network` block — but the interface name varies by hardware (e.g., `enp1s0`, `enp2s0`, `eno1`). Check with `ip link` after first boot.

> **interactive-sections:** Setting this to `[]` prevents Ubuntu 24.04 from pausing to ask for confirmation during autoinstall. Without it, the installer may wait for a keypress.

**4. Boot and install**

1. Insert the USB drive into the Beelink.
2. Boot from USB (press `F7` during POST for the one-time boot menu, or set USB as first boot device in BIOS).
3. Select the Ubuntu ISO from the Ventoy menu.
4. The installer runs unattended — no keyboard or monitor needed after this point.
5. When complete, the machine reboots. Remove the USB drive.
6. Connect over SSH from another machine: `ssh josh@<server-ip>`

> For a fully headless setup, configure the Beelink BIOS to boot from USB automatically so you never need a monitor. After install, it will boot from the NVMe on its own.

# Docker Installation

Install Docker Engine + Docker Compose v2 on Ubuntu Server 24.04 LTS.

## 1. Install Docker Engine

Official Docker repository method (not the Ubuntu `docker.io` snap):

```bash
# Remove old versions
sudo apt remove -y docker docker-engine docker.io containerd runc 2>/dev/null

# Add Docker's official GPG key
sudo apt update
sudo apt install -y ca-certificates curl
sudo install -m 0755 -d /etc/apt/keyrings
sudo curl -fsSL https://download.docker.com/linux/ubuntu/gpg -o /etc/apt/keyrings/docker.asc
sudo chmod a+r /etc/apt/keyrings/docker.asc

# Add the Docker repository
echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.asc] https://download.docker.com/linux/ubuntu \
  $(. /etc/os-release && echo "$VERSION_CODENAME") stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

# Install Docker Engine + Compose plugin
sudo apt update
sudo apt install -y docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin
```

## 2. Post-Install Configuration

### Add your user to the docker group (no `sudo` needed for docker commands)

```bash
sudo usermod -aG docker $USER
newgrp docker  # Apply immediately (or log out/in)
```

### Verify installation

```bash
docker --version
docker compose version
docker run hello-world
```

### Enable Docker to start on boot

```bash
sudo systemctl enable docker
sudo systemctl enable containerd
```

## 3. Configure Docker Daemon

Create `/etc/docker/daemon.json` for logging and storage configuration:

```bash
sudo tee /etc/docker/daemon.json > /dev/null << 'EOF'
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3"
  },
  "storage-driver": "overlay2",
  "default-address-pools": [
    {
      "base": "172.20.0.0/16",
      "size": 24
    }
  ]
}
EOF

sudo systemctl restart docker
```

This configuration:
- **Log rotation**: Caps container logs at 10MB × 3 files (prevents disk fill)
- **overlay2**: Best-performing storage driver for Ubuntu
- **Address pool**: Avoids conflicts with common LAN subnets (192.168.x.x, 10.x.x.x)

# Tailscale VPN Setup

Set up Tailscale for secure remote access to your homelab without port forwarding.

## Why Tailscale

- **Zero config VPN**: No port forwarding, no dynamic DNS, no firewall holes
- **MagicDNS**: Access services via `beelink.tailnet-name.ts.net` from any device
- **Subnet routing**: Expose your entire LAN to Tailscale-connected devices
- **Free tier**: Covers personal use (up to 100 devices, 3 users)

## 1. Install Tailscale on the Server

```bash
# One-line install
curl -fsSL https://tailscale.com/install.sh | sh

# Start and authenticate
sudo tailscale up

# Follow the URL printed to authenticate with your Tailscale account
```

## 2. Enable Subnet Routing (Optional)

Allow Tailscale-connected devices to access your entire LAN:

```bash
# Enable IP forwarding
echo 'net.ipv4.ip_forward = 1' | sudo tee -a /etc/sysctl.d/99-tailscale.conf
echo 'net.ipv6.conf.all.forwarding = 1' | sudo tee -a /etc/sysctl.d/99-tailscale.conf
sudo sysctl -p /etc/sysctl.d/99-tailscale.conf

# Advertise your LAN subnet
sudo tailscale up --advertise-routes=192.168.1.0/24 --accept-routes
```

Then approve the subnet route in the [Tailscale admin console](https://login.tailscale.com/admin/machines):

MagicDNS gives you hostnames like `beelink.tailnet-name.ts.net`:

1. Go to [Tailscale DNS settings](https://login.tailscale.com/admin/dns)
2. Enable **MagicDNS**
3. Your server is now accessible at `beelink` from any Tailscale-connected device

## 4. Install Tailscale on Your Devices

- **Windows**: Download from https://tailscale.com/download
- **macOS**: App Store or https://tailscale.com/download

## 7. Exit Node (Route All Traffic Through Server)

## FlareSolverr + Galloping Bot

The jobs stack now includes an internal-only FlareSolverr service used to pre-solve the EZLinks Cloudflare challenge for Galloping Bot.

### How it works

- The cron helper starts at 11:30 AM ET, but the FlareSolverr solve does not happen immediately.
- The actual FlareSolverr request happens when Galloping Bot begins its pre-warm window about 5 minutes before release, so for a noon release it starts around 11:55 AM ET.
- FlareSolverr requests `https://unioncountygolf.ezlinksgolf.com/index.html` and solves the interactive challenge once.
- Galloping Bot imports the returned `cf_clearance` and related cookies into its nodriver browser session.
- After that import, Galloping Bot still uses its existing browser-backed XHR flow for `GET /api/search/init`, `POST /api/search/search`, and the booking/cart endpoints.
- The first live tee-time search still waits until the release time plus `GALLOPING_SNIPE_DELAY`, so with the default 2-second delay the first real search happens at about 12:00:02 PM ET.

This is important: direct cookie replay with a plain HTTP client still gets blocked on EZLinks. The supported pattern is FlareSolverr cookie prewarm plus browser XHRs from Galloping Bot.

### Validate the helper service

```bash
docker compose -f compose/compose.jobs.yml --env-file .env up -d flaresolverr
docker compose -f compose/compose.jobs.yml --env-file .env logs -f flaresolverr
curl http://localhost:8191/
```

The readiness endpoint should return JSON similar to `{"msg":"FlareSolverr is ready!"}` if you temporarily publish the port for debugging. In the committed compose file the service stays internal-only and is reached as `http://flaresolverr:8191/v1` from the Galloping Bot container.

### Smoke-test the booking flow

```bash
docker compose -f compose/compose.jobs.yml --env-file .env run --rm galloping-bot --discover
```

If the prewarm is working, the bot should clear Cloudflare, call `GET /api/search/init`, and print the live EZLinks course list without manual intervention.

## Labs-Wiki Free-Tier URL Backfill

**Stack:** `jobs` | **Execution:** host cron wrapper | **Schedule:** hourly until complete

This helper runs the main `labs-wiki` checkout conservatively against the GitHub Models free tier. Each invocation processes at most one URL raw, prioritizes the high-value score lane first, then continues into the full remainder once the high-value queue is empty.

### How it works

- The host wrapper lives at `scripts/ops/labs-wiki-backfill.sh`.
- It reads `.env`, uses the root `labs-wiki` checkout by default, and honors `LABS_WIKI_BACKFILL_PATH` only when you intentionally want a different checkout/worktree.
- It reuses `GITHUB_MODELS_TOKEN`.
- If host `python3` is missing ingest dependencies, it bootstraps a local venv at `${HOMELAB_BASE}/data/labs-wiki-backfill/.venv`.
- Every run writes queue state under `${HOMELAB_BASE}/data/labs-wiki-backfill/`.
- If GitHub Models returns a free-tier limit signal, the wrapper sends an ntfy and exits so the next hourly cron window can retry.
- When both the high-value and full queues are empty, it sends a single completion ntfy and stays quiet on later cron runs.

### Manual status / smoke test

```bash
./scripts/ops/labs-wiki-backfill.sh --status
./scripts/ops/labs-wiki-backfill.sh --high-value
```

Use `--status` to inspect the remaining high/full queues without spending quota. `--high-value` or `--full` force a single-item run from that lane.

### Cron install

```bash
5 * * * * /opt/homelab/scripts/ops/labs-wiki-backfill.sh >> /opt/homelab/logs/labs-wiki-backfill.log 2>&1
```

That schedule runs once per hour around the clock until the queue is empty. The wrapper remains quiet after completion, so leaving the cron entry installed is safe.

## Backups

### Automated backup script

Run the backup script manually or via cron:

```bash
# Manual backup
./scripts/ops/backup.sh

# Automated daily backup at 3 AM
(crontab -l 2>/dev/null; echo "0 3 * * * /opt/homelab/scripts/ops/backup.sh >> /opt/homelab/logs/backup.log 2>&1") | crontab -
```

### What gets backed up

| Data | Method | Location |
|------|--------|----------|
| Service configs | tar.gz of `/opt/homelab/config/` | NAS or local fallback |
| Nextcloud DB | `mysqldump` from MariaDB container | NAS or local fallback |
| Immich DB | `pg_dump` from Immich PostgreSQL container | NAS or local fallback |
| Vaultwarden data | Included in config tar | NAS or local fallback |
| `.env` | Included in backup | NAS or local fallback |
| Compose files | Git (source of truth) | Git remote |

> **NAS unavailable?** The backup script falls back to `${HOMELAB_BASE}/backups/` on the local SSD with a warning. See [NAS Setup](07-nas-setup.md) to configure the NAS target later.

### What is NOT backed up

| Data | Reason |
|------|--------|
| Plex/Jellyfin metadata | Regenerated from media files; large; not critical |
| Riven VFS mount | Virtual — nothing to back up |
| Docker images | Re-pulled on deploy |
| Container logs | Ephemeral, rotated by Docker daemon |

### Restore from backup

```bash
# Stop all services
cd /opt/homelab
docker compose -f compose/docker-compose.yml --env-file .env down

# Restore configs
tar -xzf /mnt/nas/backups/homelab-config-YYYY-MM-DD.tar.gz -C /opt/homelab/

docker compose -f compose/compose.cloud.yml --env-file .env up -d nextcloud-db

sleep 5
docker compose -f compose/compose.photos.yml --env-file .env exec -T immich-postgres \
  psql -U postgres -d immich < /mnt/nas/backups/immich-db-YYYY-MM-DD.sql

# Restart everything
./scripts/ops/deploy.sh
