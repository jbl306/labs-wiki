# midudev/autoskills

## What it is

`autoskills` is a single-command (`npx autoskills`) tool from midudev that scans a project, infers the tech stack from manifests like `package.json`, Gradle files, `deno.json`, `pyproject.toml`, etc., and installs the matching AI agent skills from `skills.sh`. If Claude Code is the target, it also writes a `CLAUDE.md` summary of the installed skills. It's effectively `create-react-app` for an agent's skill library.

## Why it matters

Useful as a reference implementation for the kind of stack-aware skill installer we'd want for new project bootstraps. We don't run `autoskills` ourselves — our skill stack is curated manually under `external-powers/` and `.github/skills/` — but the detection ruleset (Next.js, Nuxt, Spring Boot, Tauri, Cloudflare Agents, Vercel AI SDK, Drizzle, Better Auth, etc.) is a good catalog of what other agent stacks consider important to specialise on.

## Architecture / Technical model

**Stack auto-detection** — Reads project manifests (`package.json`, `gradle/libs.versions.toml`, `deno.json`, `pyproject.toml`, `*.csproj`, `*.sln`, `composer.json`, `Package.swift`, etc.) to infer technologies.

> See [[automated-ai-skill-stack-installation]] for the full detection→install pipeline.

**SKILLS_MAP** — Central registry mapping detected techs → skills.sh skill IDs. ~80 entries covering frontend, backend, mobile, cloud, tooling, auth, data, AI, media. Example: `"next.js": "mstechl/nextjs"`, `"supabase": "superskills/supabase"`.

**Multi-agent target support** — Detects Claude Code, OpenCode, Kiro, and others via heuristics (presence of `.claude/`, `.opencode/`, `kiro.config.json`). Installs to correct per-agent directory.

> See [[agent-skill-routing-architecture]] for how skills are routed across agent systems.

**Interactive multi-select TUI** — Uses `@clack/prompts` to show detected skills with pre-unchecked items (already installed). Arrow keys + space to toggle. `-y` flag auto-accepts.

**Dry-run support** — `--dry-run` shows what would be installed without writing to disk. Useful for CI checks or auditing.

**Optional CLAUDE.md generation** — If `claude-code` is detected (or forced with `-a claude-code`), writes `CLAUDE.md` summarizing installed skills. Stopped being auto-generated by default in v0.2.6; behavior varies by version.

**Skills fetched from skills.sh** — Public registry at [skills.sh](https://skills.sh). No auth required. Skills are markdown files with frontmatter + instructions.

## How it works

### Detection phase (runs locally, zero network)

1. **Workspace resolution** — Walks up from cwd to find project root (presence of `package.json`, `.git`, `go.mod`, etc.). Also checks `deno.json`, `deno.jsonc`, `bun.lock`, `pnpm-lock.yaml`, `composer.json`.

2. **Manifest parsing** — For each detected root:
   - **JS/TS**: `package.json` dependencies + devDependencies + config files (`next.config.*`, `nuxt.config.*`, `vite.config.*`, `tailwind.config.*`, `vercel.json`, `.vercel/`, `wrangler.toml`, etc.)
   - **Gradle**: `build.gradle`, `build.gradle.kts`, `settings.gradle`, `gradle/libs.versions.toml` — detects Android, Kotlin Multiplatform, Spring Boot
   - **Python**: `pyproject.toml`, `requirements.txt`, `setup.py` — detects Django, FastAPI, Flask, Pydantic, pytest, etc. (added v0.2.6)
   - **.NET**: `*.csproj`, `*.sln`, `global.json`, `NuGet.Config` — detects ASP.NET Core, Blazor, Minimal API (added v0.2.6)
   - **Go**: `go.mod`, `go.work`
   - **Dart/Flutter**: `pubspec.yaml` with `flutter:` key (added v0.2.6)
   - **Swift**: `Package.swift` — detects SwiftUI, Xcode build, Swift concurrency, Core Data (added v0.2.4)
   - **PHP**: `composer.json`, `wp-config.php`, Laravel detection via `artisan`, `app/` structure, `bootstrap/app.php` (added v0.2.6)
   - **Ruby**: Gemfile (added v0.2.6)
   - **Electron**: `electron` package, `electron-builder.yml`, `forge.config.js`, `electron-vite.config.ts` (added v0.2.6)
   - **Tauri**: `@tauri-apps/*` packages, `src-tauri/tauri.conf.json`

3. **Web frontend signal scan** — If no framework detected, scans for `.html`, `.css`, `.scss`, `.vue`, `.svelte`, `.jsx`, `.tsx`, `.twig`, `.blade.php` — installs generic frontend design / accessibility / SEO skills.

4. **Combo detection** — Multi-signal patterns: Next.js + Supabase, Next.js + Vercel AI SDK, React + shadcn/ui, Tailwind + shadcn, Expo + Tailwind, React Hook Form + Zod, GSAP + React, Cloudflare + Vite, Node + Express.

### Installation phase (network-dependent)

1. **Skill selection** — For each detected tech, looks up skill ID in `SKILLS_MAP`. Fetches from `skills.sh` API.

2. **Multi-select TUI** — Shows list with checkboxes. Already-installed skills pre-unchecked. User can toggle, or `-y` accepts all.

3. **Parallel install** — Fetches skills concurrently. Progress bar shows live status.

4. **Write to disk** — For each skill:
   - `.claude/skills/<skill-name>/SKILL.md` (Claude Code)
   - `.opencode/skills/<skill-name>/SKILL.md` (OpenCode)
   - `.kiro/skills/<skill-name>/SKILL.md` (Kiro)
   - Other agents: varies

5. **CLAUDE.md generation** (optional) — If Claude Code detected, reads all `.claude/skills/*/SKILL.md`, extracts titles and descriptions, writes summary to `CLAUDE.md`.

6. **Summary output** — Shows installed skills, skipped (already present), errors. Structured output with colors + icons.

## Setup

```bash
# Run in project root
npx autoskills

# Or install globally
npm install -g autoskills
autoskills

# Preview without installing
npx autoskills --dry-run

# Auto-accept detected skills
npx autoskills -y

# Force Claude Code target
npx autoskills -a claude-code
```

## Integration notes

**Not for direct adoption** — Our skill stack is curated manually in `external-powers/` and `.github/skills/`, tracked in git. `autoskills` pulls from `skills.sh`, which is a different supply chain.

**Useful as reference**:
- **Detection ruleset catalog** — The `SKILLS_MAP` is a survey of "what does the agent skill ecosystem think matters per stack." Useful when curating our own skill library.
- **Multi-ecosystem detection patterns** — Gradle, `.csproj`, `pyproject.toml`, `composer.json` detection code is reusable for building custom stack-aware tooling.
- **Combo detection logic** — Inspires similar multi-signal detection for `context-doc-factory` or homelab service recommendations.

**Alternative use case: CI check** — `npx autoskills --dry-run` in CI to flag when a project's tech stack diverges from its installed skills.

## Caveats / Gotchas

- **License: CC BY-NC 4.0** — Non-commercial. Reusing detection code or SKILLS_MAP in commercial workspace tools requires explicit permission or license change.
- **Requires Node.js >= 22** — Uses Node 22-specific APIs (e.g. `node --test` for tests).
- **Build changed in v0.2.7** — Now compiles TypeScript to `dist/` and ships `dist/main.js`. Older versions ran `.ts` directly via tsx. Package.json `bin` points to `index.mjs`, which imports `dist/main.js` (with fallback to `main.ts` in dev).
- **CLAUDE.md generation behavior changed** — v0.2.6 stopped auto-generating CLAUDE.md by default in some flows. Check `--help` for current behavior.
- **Stale SKILLS_MAP entries** — Some skills may have moved/renamed in skills.sh registry. Installation errors are surfaced in summary output.
- **No lockfile tracking** — `autoskills` doesn't write a lockfile of installed skills (unlike npm/poetry). Repeat runs may re-fetch skills.
- **No uninstall command** — Removing skills requires manual `rm -rf .claude/skills/<skill-name>` or equivalent.
- **Detection false positives** — E.g. having `typescript` in devDependencies doesn't guarantee the project uses TypeScript for source code. Detection heuristics are coarse-grained.
