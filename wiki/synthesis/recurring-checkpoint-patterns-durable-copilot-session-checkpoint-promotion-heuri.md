# Recurring checkpoint patterns: Durable Copilot Session Checkpoint Promotion, Heuristic-Based Classification of Session Checkpoints, Auto-Ingest Pipeline for Wiki Markdown Processing

## Question

What recurring decisions, fixes, and durable patterns appear across the 30 session checkpoints in this cluster, especially around Durable Copilot Session Checkpoint Promotion, Heuristic-Based Classification of Session Checkpoints, Auto-Ingest Pipeline for Wiki Markdown Processing?

## Summary

Across 30 checkpoints, the durable pattern is the same loop applied to itself: route raw inputs through source-aware lanes, suppress non-durable content with heuristics, compile only durable summaries through GitHub Models, then feed the compiled artifacts back into the persistent store (wiki + MemPalace). The recurring decisions are about what to keep, what to ingest only once, and what to mirror across systems—not about the LLM call itself.

## Analysis

The throughline across this 30-checkpoint cluster is a layered guard around the single line of code that costs money: the GitHub Models call inside `auto_ingest.py`. Every other recurring fix in the cluster either *prevents* that call from firing on low-value content (Heuristic classification), *deduplicates* what reaches it (SHA-256 source-hash gate, mempalace curator hashes), or *amortizes* the result across surfaces (MemPalace re-ingest, agent persona docs).

The second persistent decision is **routing by source class**. Copilot session checkpoints are the dominant input volume but the lowest information density per byte, so they get a 'light' lane with tighter caps. Standard URL/text gets the default lane. Image-bearing content gets vision. The route classifier and inflight suppression set together stop the pipeline from compiling the same checkpoint twice concurrently—a real bug encountered when the watcher fired on partial writes.

Third, the cluster repeatedly converges on **distilled-only promotion**. Conversational memory (raw checkpoint files) is treated as ephemeral; only the curator's hashed summary crosses into wiki/raw/. This is the operationalization of Karpathy's 'compile once' principle: the wiki is the cache of compiled knowledge, not a copy of session state. MemPalace closes the loop by re-ingesting compiled wiki pages so search and graph operations can happen locally without re-prompting a model.

Fourth, **observability is non-negotiable**. Every fix in the cluster shipped with ntfy notifications, log lines, or a metrics counter. When notifications themselves broke (UTF-8 emoji encoding), that was treated as a P0 because operators lose the loop closure signal.

The Custom Copilot CLI Agents work is the meta-application: instead of the wiki documenting the pipeline, agents *operate* it. The persona files (AGENTS.md tables, 13-stack homelab-ops, deployment archetype, ML pipeline, knowledge curator) make the curation loop self-applying—an agent can re-trigger ingest, validate it, and ship a checkpoint that itself gets routed back through the same pipeline.

1. **Heuristic classification is more valuable than smarter prompting—suppressing extraction on planning-only checkpoints saves more credits than any prompt optimization in the cluster.** — supported by [[Heuristic-Based Classification of Session Checkpoints]], [[Auto-Ingest Pipeline for Wiki Markdown Processing]]
2. **The SHA-256 hash idempotency gate appears in three independent places (curator, ingest, MemPalace re-import) because each surface re-derived the same lesson: 'never recompute what you've already compiled.'** — supported by [[Durable Copilot Session Checkpoint Promotion]], [[Auto-Ingest Pipeline for Wiki Markdown Processing]]
3. **Source-aware lane routing (light/default/vision) is the only place GPT-4o-class compute is metered—everything upstream and downstream is local Python or hash math, which is why the Karpathy compile-once mapping is exact rather than aspirational.** — supported by [[Durable Copilot Session Checkpoint Promotion]], [[Auto-Ingest Pipeline for Wiki Markdown Processing]]
4. **Custom CLI agents are how the curation loop closes on itself: an agent can ingest, classify, and validate a checkpoint—then write a new checkpoint that re-enters the same pipeline, which is the observable definition of a durable workflow.** — supported by [[Custom Copilot CLI Agents]], [[Durable Copilot Session Checkpoint Promotion]]
