# Recurring checkpoint patterns: Parallel Agent Coordination in ML Sprint Implementation, Calibration Leakage Mitigation in ML Model Training, Data Source Expansion for NBA ML Prediction Platform

## Question

What recurring decisions, fixes, and durable patterns appear across the 41 session checkpoints in this cluster, especially around Parallel Agent Coordination in ML Sprint Implementation, Calibration Leakage Mitigation in ML Model Training, Data Source Expansion for NBA ML Prediction Platform?

## Summary

Across 41 NBA-ML / homelab checkpoints the recurring decisions form a tight ML-platform loop: split data into temporally-honest train/calibration sets, expand features in *phases* with explicit validation gates, run multiple feature-and-fix workstreams in parallel via shared git, and deploy as Docker stacks scheduled by Ofelia on the Beelink. The durable lesson is that the platform's reliability comes from *gates between phases*—calibration leakage prevention, walk-forward CV, sprint progress trackers—rather than from any single model improvement.

## Analysis

This is the largest cluster in the wiki and it documents the operational backbone of the NBA ML Engine. The recurring pattern is **gated phases**: every concept here describes a workflow that refuses to advance past an explicit checkpoint until a measurable condition holds. Sprint coordination won't merge before tests pass; calibration won't fit before training is done; data-source expansion won't promote phase N+1 before phase N has nulls and coverage validated; deployment won't release before migrations and container builds succeed.

Calibration leakage is the most expensive of the four because the failure is silent. The Sprint 55 fix—80/20 split and `cv='prefit'`—works precisely because it prevents the leakage at the data-handling layer instead of trying to detect it later. The incidental fixes from the same checkpoint (XGBoost early-stopping moved to `set_params`, feature medians persisted as JSON) follow the same pattern: keep the contract enforceable at training time, not inferred at prediction time.

Data-source expansion is the largest scope-control problem. The phase split (1A/B/C → 2 → 3/4) is what makes the platform's growth tractable. Each phase paid the cost of integration and validation in isolation—position backfill, then starter data, then game-advanced rolling features—so when the NBA API outage hit, only the active phase had to fall back to CDN proxies. The 0.6s nba_api delay and ~20 req/min Basketball Reference cap are budget facts the phasing has to respect.

Parallel agent coordination is how this scales beyond what one operator can hold in head: four agents on Sprint 55 worked in parallel on cleanup, calibration fixes, edge optimization, and training pipeline, synchronized only by `git add -A`, the progress tracker, and the agreed-on phase gates. The recurring pitfalls—agents overwriting each other, Alembic auto-generation drifting the schema—were both addressed by *more* discipline (frequent commits, hand-edited migrations) rather than fewer agents.

The homelab deployment stack is the load-bearing element under all of the above. TimescaleDB + MLflow + FastAPI + Streamlit + Ofelia in five containers on a single Beelink is the smallest configuration that supports the gated workflow without introducing cloud-billing complexity. The recurring failure here is environment loading—`.env` not found at compose-up time crashes the database, which is why the homelab archetype always invokes `docker compose --env-file .env -f compose/...`.

## Key Insights

1. **The platform's robustness comes from *gates between phases*, not from any model improvement: every recurring fix added a contract that prevents the prior failure mode from re-occurring (calibration split, phase validation, pre-merge tests).** — supported by [[Calibration Leakage Mitigation in ML Model Training]], [[Data Source Expansion for NBA ML Prediction Platform]], [[Parallel Agent Coordination in ML Sprint Implementation]]
2. **Parallel-agent sprints scale only because the coordination primitives are deliberately low-tech: feature branch + `git add -A` + a progress tracker + hand-edited migrations. Higher-tech coordination (auto-generated migrations, lockfiles per agent) was tried and rejected.** — supported by [[Parallel Agent Coordination in ML Sprint Implementation]]
3. **Data-source expansion respects external API budgets (0.6s nba_api delay, 20 req/min Basketball Reference) as first-class design constraints—phase boundaries match what one rate budget can ingest in a sitting.** — supported by [[Data Source Expansion for NBA ML Prediction Platform]]
4. **The single-Beelink deployment is intentionally the smallest configuration that fits the gated workflow; horizontal scale is deferred until a phase gate would actually be blocked by it.** — supported by [[Homelab Server Deployment Architecture for NBA ML Platform]]

## Open Questions

- What is the right re-validation cadence for calibration when feature distributions drift mid-season—do we re-split and re-fit on every nightly retrain, or only when a drift detector trips?
- Should phased data-source expansion adopt a formal entry/exit criteria spec per phase (like a release checklist) so non-author agents can ship a phase without re-deriving the gates?
- When the Beelink starts hitting CPU/RAM ceilings, which subsystem should split off first—MLflow tracking, TimescaleDB write load, or the dashboard—and what's the trigger metric?
